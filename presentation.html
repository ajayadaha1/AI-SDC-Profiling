Problem Statement AI Failure Profiling

Why This Matters

When a failing CPU hits the bench, engineers guess which commands to run.

🔴 Today — Reactive Debug

  • Engineer manually interprets vague symptoms
  • Runs broad sweep of AFHC/ANC commands
  • Most commands pass — wasted hours
  • Knowledge trapped in individual heads
  • Turnaround: hours to days per unit
→

🟢 Proposed — Predictive Debug

  • Input symptoms → model cross-references 1000s of cases
  • Returns ranked commands by failure probability
  • Hits failure on first or second attempt
  • Institutional knowledge codified & amplified
  • Turnaround: minutes per unit
The Core Question AI Failure Profiling

What Are We Solving?

"Given a new CPU presenting symptoms S = {MCE code, bank ID, error type, thermal state, voltage, boot stage, stepping, SKU, …}, which AFHC/ANC commands C₁, C₂, …, Cₖ have the highest probability of reproducing the failure?"

This is a Learning-to-Rank Problem

  • Train a model to score every candidate command against the symptom vector
  • Sort by predicted failure probability — present top-K to engineer
  • Failure patterns repeat: Bank 10 MCE under thermal stress in 2026 ≈ same fault class seen in 2024

Key Interaction Features

Why LightGBM?

  • Best-in-class for tabular / mixed-type data
  • Handles missing values natively
  • Fast training + built-in feature importance
  • Class-weighted loss handles imbalanced PASS/FAIL
  • Optuna auto-tunes hyperparameters

How We Train & Measure

Data Splitting

Train (70%)

CPUs debugged before cutoff T

Test (15%)

CPUs after T+Δ — held-out, never touched during training

Risk Mitigation AI Failure Profiling

Risks & Mitigations

RiskImpactMitigation
Insufficient FAIL labels
Most commands pass → sparse signal		 High Class-weighted loss, SMOTE, focal loss. Collect "near-miss" anomalies.
Novel failure mode
No historical precedent		 Medium Fall back to FAISS retrieval (Stage 1). Flag low-confidence for manual review.
Data quality issues
Inconsistent logging, missing fields		 Medium Phase 0 data audit. LightGBM handles missing data natively. Robust cleaning pipeline.
Concept drift
New steppings have different patterns		 Medium Weekly retraining, drift detection, stepping-aware features + transfer learning.
Engineer distrust
"Black box" resistance		 High SHAP explanations per prediction. Show similar historical cases. "Recommend" not "direct."
